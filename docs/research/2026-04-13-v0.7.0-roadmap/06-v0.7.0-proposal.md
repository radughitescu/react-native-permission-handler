# v0.7.0+ Roadmap Proposal

Based on five-agent research (competitor survey → pain mining → cold-integration friction → library advocate triage → devil's advocate review).

## Top-line recommendation

**v0.7.0 is the "finish what we shipped" release.** Wire up the half-connected v0.6.0 features, document the full API surface, and ship one high-impact Android normalization layer. Every item in the recommended cut already has a home in the existing architecture — no new primitives, no native code, no persistence.

The single biggest roadmap signal across all sources is **Android post-dialog status normalization** (6 compounded upstream tickets). The single biggest user-reach win is **the README rewrite** — our cold-integration persona walked away thinking the library was 60% of what actually shipped.

## Recommended v0.7.0 cut line

**Ship items 1–10 in v0.7.0.** Rationale:

- **7 of 10 are Small effort.** The release is shippable in a single focused push.
- **Items 1 + 2 are the headliners.** Android normalization is the most evidence-backed feature on the list. README rewrite is the force multiplier that makes every other shipped feature actually reach users.
- **Items 3, 4, 5 finish v0.6.0.** `requestFullAccess` without `renderLimited` is a half-built feature. Optional config fields and the Android 16 timeout default are both one-line type/constant changes that unblock real flows today.
- **Items 6–10 are all S and all under 100 lines each.** Adding them to v0.7.0 costs almost nothing and removes five distinct cold-integration frictions in one release.
- **Cutting here keeps the release thematic:** "every v0.6.0 feature now actually reaches users, plus the biggest single reliability fix."

## Deferred to v0.8.0+

- **Item 11 (`openSettings` deep-linking)** — M effort, iOS URL schemes are unofficial and brittle. Worth doing, but wants its own release so we can ship a clean opt-out and document the fragility. Good v0.8.0 anchor.
- **Item 12 (preflight diagnostic)** — M effort, requires careful first-mount probe logic and an opt-out path. Also a natural v0.8.0 fit alongside `openSettings` deep-linking under a "DX polish" release theme.
- **Conditional items from devil's advocate review:**
  - `dependsOn` sequential graph → ship narrowly as part of item 7 (bundles); revisit as a general primitive only if bundles aren't enough.
  - Expo config plugin for Bluetooth/Location manifest setup → investigate what the existing Expo auto-discovery already covers; if a gap remains, scope a v0.8.0 task.
  - `dismissPolicy` with TTL → tied to a future pluggable storage primitive; defer until we commit to storage as a first-class concept (probably never — apps can do this in 10 lines).
  - `PermissionsWall` component → ship as a docs recipe, not a component. Roll into item 2.
  - iOS limited photo picker re-prompt (Expo-only slice) → already covered by item 3's Expo branch.

## Cut from consideration

Killed by the devil's advocate and staying dead:

- **One-time grants auto-revoke detection** — no native signal available; heuristic synth events would fire on every backgrounding. Footgun factory.
- **iOS "Allow Once" vs "Allow While Using" status** — requires native bridge; violates the pure-JS constraint.
- **`PermissionDescriptor` refactor for settings-only permissions** — breaking change to core type for a single-ticket request. Could be a v1.0 redesign topic, not v0.7.0.
- **First-ask / `undetermined` tracking with pluggable storage** — adds a new status value and a persistence primitive for a synthetic distinction most apps don't care about.
- **Android 13 cold-start resume after `openSettings` kills the process** — complexity budget is massive; user base is a single OS-version-specific flow; document it as a known limitation.
- **Multi-target iOS (App Clip / share extension)** — lives in the RNP Podfile, below our layer.
- **Android `checkMultiple` probe helper** — the fix (auto-calling `request()` to probe) is exactly the antipattern the library exists to prevent.

## Open questions for the user

Four decisions before plan-and-execute cycles begin:

1. **Does "v0.7.0 is the finish-what-we-shipped release" land?** Or do you want v0.7.0 to be narrower (just items 1–2) and defer the rest to v0.7.1+?
2. **Item 1 (Android normalization) — opt-in flag or default on?** Devil's advocate argued for opt-in (`createRNPEngine({ normalizeAndroid: true })`) because heuristics can misfire on edge cases. Default-on is more impactful but riskier. Your call.
3. **Item 2 (README rewrite) — how aggressive?** Options: (a) plain README overhaul, (b) README + a dedicated `/docs` folder with per-API pages, (c) README + a docs site (docusaurus/vocs). (a) is ~1 day, (c) is a separate project.
4. **Do you want a separate "DX polish" v0.8.0 right after, bundling items 11–12 + the conditional Expo config plugin work?** Or roll them into v0.7.0 and delay the release?

Once these are answered, each v0.7.0 item gets its own brainstorm → spec → plan → execute cycle.
